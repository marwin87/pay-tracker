# PWA Installability Implementation Plan

## Overview

Implement FR-013: make Pay Tracker installable as a PWA on mobile and desktop. The three
browser-required ingredients are a valid web app manifest, app icons (192×192 and 512×512),
and HTTPS. We add a minimal service worker (fetch passthrough) for cross-browser
compatibility (Firefox, older Safari). No offline caching, no custom install UI — the
browser's native install prompt handles UX.

## Current State Analysis

The frontend has zero PWA infrastructure:
- No `app/manifest.ts` or `manifest.json`
- No service worker
- No app icons (only Next.js default SVGs in `public/`)
- No PWA-related packages installed
- `next.config.ts` has only `output: "standalone"` — no headers configured
- `src/app/layout.tsx` exports a `metadata` object with title/description only

Next.js 16.2.9 ships built-in App Router support for web app manifests via
`app/manifest.ts` (returns `MetadataRoute.Manifest`). No external library is needed.
The bundled docs at `node_modules/next/dist/docs/01-app/02-guides/progressive-web-apps.md`
are the authoritative reference.

## Desired End State

- Visiting the app in Chrome/Edge/Firefox/Safari triggers a browser-native "install" /
  "Add to Home Screen" prompt.
- The installed app opens in standalone mode (no browser chrome), with a white splash
  background and blue theme bar.
- Lighthouse PWA installability audit passes (manifest valid, icons present, SW registered).
- The app icon on home screen shows the placeholder Pay Tracker icon (to be replaced with a
  real brand icon in a later change).

### Key Discoveries

- `app/manifest.ts` is the Next.js 16 App Router mechanism — Next.js automatically serves it
  at `/manifest.webmanifest` and injects `<link rel="manifest">` into the page head.
  No `<link>` tag needed in `layout.tsx`: `frontend/src/app/layout.tsx:17`
- `themeColor` in the Next.js `Metadata` type applies `<meta name="theme-color">` globally.
  It belongs in `layout.tsx`'s `metadata` export, not the manifest (both are needed —
  manifest for install; meta tag for browser chrome while browsing).
- Service worker must be in `public/sw.js` (served as a static file). Next.js does not
  process SW files through its module bundler — it must be plain JS.
- SW registration must happen in a `'use client'` component (needs browser APIs).
  The registration component is added to `layout.tsx`'s body, outside any auth/locale
  providers.
- `next.config.ts` currently uses `export default` ESM syntax — any headers config must
  follow the same pattern (not `module.exports`).

## What We're NOT Doing

- **No offline caching.** The SW is a fetch passthrough only — it does not cache any
  assets. Offline support is a separate concern.
- **No push notifications.** FR-012 (email reminders) is nice-to-have and email-based.
  Browser push is a future feature.
- **No custom install banner.** Browser's native `beforeinstallprompt` / "Add to Home
  Screen" flow is sufficient. No custom React UI.
- **No HTTPS configuration.** HTTPS is a deployment concern noted in the PRD Open
  Questions; it is out of scope for this plan.
- **No real brand icon.** Placeholder PNGs are the deliverable here; a real icon is a
  design task.

## Implementation Approach

Use Next.js 16's native App Router conventions throughout:
1. `app/manifest.ts` for the web app manifest (auto-served, auto-linked).
2. Placeholder PNG icons in `public/` generated by any available tool (ImageMagick,
   browser-based favicon generator, or a one-off Node script).
3. A minimal `public/sw.js` with only a fetch passthrough event handler.
4. A small `'use client'` component that registers the SW on mount — imported in
   `layout.tsx` as a leaf node.
5. Security headers for `sw.js` added to `next.config.ts`.

## Phase 1: Web App Manifest and Icons

### Overview

Create the web app manifest, app icons, and add `themeColor` to the root metadata. This
phase alone makes the app installable in Chrome and Edge; it is also the prerequisite for
Phase 2.

### Changes Required

#### 1. Web App Manifest

**File:** `frontend/src/app/manifest.ts`

**Intent:** Export the web app manifest so Next.js auto-serves it at
`/manifest.webmanifest` and injects `<link rel="manifest">` into every page.

**Contract:** The default export is a function returning `MetadataRoute.Manifest` with
these fields:
- `name: "Pay Tracker"`
- `short_name: "Pay Tracker"`
- `description: "Household bill tracking made simple"`
- `start_url: "/"`
- `display: "standalone"`
- `background_color: "#ffffff"`
- `theme_color: "#2563eb"`
- `icons`: two entries — `{ src: "/icon-192.png", sizes: "192x192", type: "image/png" }`
  and `{ src: "/icon-512.png", sizes: "512x512", type: "image/png" }`

#### 2. App Icons

**Files:** `frontend/public/icon-192.png`, `frontend/public/icon-512.png`

**Intent:** Provide the mandatory icon assets referenced by the manifest. These are
placeholder graphics to be replaced with a real brand icon later.

**Contract:** Two PNG files at exactly 192×192px and 512×512px. Content: any recognisable
placeholder (e.g., a blue circle with "PT" initials on white background). Generate using
ImageMagick, a browser-based favicon generator (e.g., realfavicongenerator.net), or a
one-off Node.js script using the `canvas` package. The file names and sizes must match the
manifest exactly.

#### 3. Root Layout — themeColor Metadata

**File:** `frontend/src/app/layout.tsx`

**Intent:** Extend the existing `metadata` export to include `themeColor` so the browser
chrome (address bar, status bar on Android) adopts the app's brand blue while the user
browses the site (not just after install).

**Contract:** Add `themeColor: "#2563eb"` to the existing `export const metadata: Metadata`
object. No other layout changes in this phase.

### Success Criteria

#### Automated Verification

- TypeScript compiles without error: `cd frontend && npx tsc --noEmit`
- ESLint passes: `cd frontend && npm run lint`
- Manifest is served correctly: `curl http://localhost:3010/manifest.webmanifest` returns
  JSON with `name`, `icons`, and `display: standalone`
- Icons are reachable: `curl -I http://localhost:3010/icon-192.png` returns `200 OK` with
  `content-type: image/png`

#### Manual Verification

- Open Chrome DevTools → Application → Manifest; all fields populate without errors
- No "installability" warnings in DevTools for missing icons or missing `display`
- Browser address bar / tab shows the blue theme color on the site

**Implementation Note:** After Phase 1 automated verification passes, pause for manual
DevTools check before proceeding to Phase 2. Phase blocks use plain bullets — the
corresponding `- [ ]` checkboxes for these items live in the `## Progress` section.

## Phase 2: Service Worker and Registration

### Overview

Add a minimal fetch-passthrough service worker and register it from a client component
wired into the root layout. Add the required security headers for `sw.js` in
`next.config.ts`. This phase satisfies the SW requirement for Firefox and strict PWA
audits.

### Changes Required

#### 1. Service Worker

**File:** `frontend/public/sw.js`

**Intent:** Provide a registered service worker so the app passes browser installability
checks that require a SW with a fetch handler. No caching — every request goes to the
network.

**Contract:** Plain JavaScript file (not TypeScript, not bundled). Contains exactly one
event listener: `self.addEventListener('fetch', (event) => { event.respondWith(fetch(event.request)); })`.
No install or activate handlers needed for a passthrough SW.

#### 2. SW Registration Component

**File:** `frontend/src/components/pwa-register.tsx`

**Intent:** Register the service worker on the client side once the component mounts.
Must be a `'use client'` component because SW registration requires browser APIs.

**Contract:** A React component that runs `navigator.serviceWorker.register('/sw.js', { scope: '/', updateViaCache: 'none' })` inside a `useEffect` on mount, guarded by
`'serviceWorker' in navigator`. Returns `null` — renders nothing. Export as default.

#### 3. Root Layout — Mount PwaRegister

**File:** `frontend/src/app/layout.tsx`

**Intent:** Mount the `PwaRegister` component so SW registration fires on every page load.

**Contract:** Import `PwaRegister` from `@/components/pwa-register` and render it inside
`<body>` as a sibling to the existing providers — place it before `<AuthProvider>` so it
is outside the auth render tree and registers regardless of auth state. No other changes.

#### 4. Security Headers for sw.js

**File:** `frontend/next.config.ts`

**Intent:** Add the recommended headers for the service worker route so it is served with
the correct `Content-Type` and never cached by the browser (stale SW is a common source
of debugging pain).

**Contract:** Extend `NextConfig` with an `async headers()` method that returns a headers
rule for `source: '/sw.js'` with:
- `Content-Type: application/javascript; charset=utf-8`
- `Cache-Control: no-cache, no-store, must-revalidate`

The existing `output: "standalone"` stays unchanged. The file uses ESM (`export default`)
— the headers method must follow the same syntax.

### Success Criteria

#### Automated Verification

- TypeScript compiles without error: `cd frontend && npx tsc --noEmit`
- ESLint passes: `cd frontend && npm run lint`
- SW is served with correct headers: `curl -I http://localhost:3010/sw.js` returns `200`
  with `Content-Type: application/javascript` and `Cache-Control: no-cache`

#### Manual Verification

- Chrome DevTools → Application → Service Workers: SW is registered, status is "activated
  and running"
- Chrome DevTools → Application → Manifest: "Installability" section shows no errors
- Chrome address bar shows the install icon (⊕ or similar) — clicking it triggers the
  native install prompt
- On a mobile device (or DevTools device emulation): "Add to Home Screen" works; installed
  app opens in standalone mode with no browser chrome
- Lighthouse → PWA audit: "Installable" check passes

**Implementation Note:** After Phase 2 automated verification passes, pause for manual
install test before closing the change. The install test on a real or emulated mobile
device is the definitive acceptance check.

## Testing Strategy

### Automated

- TypeScript (`npx tsc --noEmit`) and ESLint (`npm run lint`) after each phase
- `curl` checks for manifest, icons, and SW headers against the running dev server

### Manual Testing Steps

1. Start dev server: `cd frontend && npm run dev` (or `docker compose up`)
2. Open Chrome → DevTools → Application → Manifest: verify all fields, no errors
3. DevTools → Application → Service Workers: verify SW registered, activated
4. DevTools → Lighthouse → Progressive Web App: run audit, confirm "Installable" passes
5. In Chrome address bar, click the install icon — confirm native install prompt appears
6. Install the app — confirm it opens in standalone mode (no browser toolbar)
7. On Android (physical or emulated): "Add to Home Screen" flow works end-to-end
8. On iOS Safari (physical or emulated): "Share → Add to Home Screen" works; icon appears
   on home screen

## Migration Notes

No database changes. No backend changes. Frontend-only additions.

The `output: "standalone"` setting in `next.config.ts` is preserved — the SW file in
`public/` is included in the standalone output by Next.js.
